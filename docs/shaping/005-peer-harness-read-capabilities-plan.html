Backlog 005 / Elixir Review Tools Parity

Make peer harness read authority explicit before adding read tools.

The Rust peer-harness path can run arbitrary CLIs, but the old Elixir tool layer still owns GitHub and local repo reads. This slice makes read capability boundaries typed profile data and execution-plan evidence, without adding a tool framework.

Chosen Design

Add one small capability struct. `PeerHarnessCapabilities` records `local_repo_read` and `github_read` booleans on profiles and plans.

The current Pi, Goose, OpenCode, OMP, and fixture profiles declare both false; prompts tell reviewers not to claim external reads.

Stop rule: no live GitHub calls, provider spend, MCP/tool runtime, or default promotion in this slice.

Outcome

Every peer harness profile and generated execution plan is inspectable for local repo and GitHub read authority before an agent is launched.

Standards

Keep `cerberus-core` source-agnostic. Keep subprocess semantics in adapter/CLI. Model capability boundaries as exact data, not prose-only warnings.

Proof

Schema/profile tests, prompt/plan CLI QA, validation commands, full gates, and fresh critics focused on overengineering and false security claims.

Verification System

Claim: peer harness read authority is visible and validated before execution.

Falsifier: a checked profile or generated plan omits read capability data, or the rendered prompt allows claims based on unavailable local/GitHub reads.

Driver: render a Pi execution plan and prompt without provider spend, validate both profile fixtures and the generated plan, and inspect the prompt boundary lines.

Evidence: `tmp/peer-harness-read-capabilities-2026-06-19/` plan, prompt, artifact, summaries, and hashes.

Critic Focus

Attack schema churn, capability overclaiming, prompt-only safety, and whether this should have been a future tool system instead of profile data.

1. Docs

Document that current peer harness profiles are prompt-contained and have no local repo or GitHub read authority.

2. Red

Add tests expecting profile fixtures, generated execution plans, and rendered prompts to expose `local_repo_read: false` and `github_read: false`.

3. Green

Add the typed capability struct, pass it through plan generation, update fixtures, and keep backward-compatible defaults for older v3 packets.

4. Gates

Run focused schema/CLI tests, no-spend CLI QA, validation commands, full workspace and legacy gates, then fresh artifact-only review.
