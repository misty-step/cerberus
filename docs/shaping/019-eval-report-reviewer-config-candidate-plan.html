Backlog 019

Turn eval winners into sandbox reviewer config candidates.

The next bridge is not a default change. It is a strict, inspectable packet generator: only fully passing live eval groups whose embedded artifacts regrade against the supplied suite and match live transcript evidence can become sandbox-only `ReviewerConfigPacket.v1` candidates, and import remains dry-run rejected until approval exists.

Outcome

`cerberus-cli propose-reviewer-config` reads a report, matrix, suite, and eval evidence directory, selects a passing live harness/model group with full suite coverage and matching transcript artifact envelopes, writes a validated packet, and lets the existing import dry-run path compare it to the baseline.

Chosen Design

Core owns winner selection and packet construction. CLI owns file parsing and output. No production defaults change, and no provider spend occurs.

Proof Surface

  • Focused core tests for candidate and refusal paths.
  • CLI QA from live fixture report to packet to import dry-run report.
  • Negative QA over provider-gated unavailable report.
  • Full Rust, shell, Node, and legacy Elixir gates.

Stop Conditions

  • Stop if offline `warn` cells can promote.
  • Stop if a packet validates without preserving report/matrix evidence.
  • Stop if this requires prompt synthesis or default mutation.

Verification System

FieldContract
ClaimA passing live eval group can produce a schema-valid sandbox candidate packet.
FalsifierOffline, unavailable, degraded, failed, truncated, hand-edited pass, or transcript-mismatched reports produce a packet; validation or dry-run import fails unexpectedly.
Drivercerberus-cli propose-reviewer-config --report ... --matrix ... --suite ... --evidence-dir ... --out ...
GraderPacket validation, dry-run import report validation, and focused Rust assertions over selection gates.
Evidencetmp/reviewer-config/eval-live-peer-candidate.json and import report.
CadenceAfter eval reports, before Daedalus import or default review.

Delete-First

Requirement questioned: do we need a new promotion workflow?

Deleted or simplified: reuse `ReviewerConfigPacket.v1` and `import-reviewer-config --dry-run`; add only a report-to-packet bridge.

Only then automated: hand-authored packet JSON is brittle and bypasses eval evidence.

Adversarial Review Focus

  • Can weak eval evidence sneak into candidate generation?
  • Can edited report fields bypass artifact regrading?
  • Can offline transcripts be edited around mode/status fields?
  • Does provider/model formatting match packet validation?
  • Does the CLI leave partial output after refusal?
  • Is the core interface deep enough or just a shallow wrapper?

Acceptance Commands

cargo run --locked -p cerberus-cli -- propose-reviewer-config \
  --report tmp/evals/live-peer/report.json \
  --matrix fixtures/evals/harness-model-live-peer-matrix.json \
  --suite fixtures/evals/reviewer-harness-live-peer-smoke.json \
  --evidence-dir tmp/evals/live-peer \
  --out tmp/reviewer-config/eval-live-peer-candidate.json

cargo run --locked -p cerberus-cli -- validate-reviewer-config \
  tmp/reviewer-config/eval-live-peer-candidate.json

cargo run --locked -p cerberus-cli -- import-reviewer-config \
  tmp/reviewer-config/eval-live-peer-candidate.json \
  --dry-run \
  --out tmp/reviewer-config/eval-live-peer-import-report.json
