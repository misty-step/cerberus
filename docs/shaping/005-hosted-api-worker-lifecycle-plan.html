Move One Hosted Review from Queued to Completed in Rust

Target outcome: backlog 005 has a no-network worker lifecycle proof. A queued hosted API review is acquired from explicit PR context and diff, reviewed by the Rust engine, written back as a completed status with a validated ReviewRunArtifact.v1, and read through the existing local HTTP fixture.

Chosen design: add a narrow worker fixture command. Keep queue semantics in the adapter, file/CLI wiring in cerberus-cli, and avoid schedulers, live GitHub calls, provider spend, or Elixir deletion.

Quality System

Proof: failing tests, schema validation, loopback GET replay, token-leak grep, full gates, fresh critic.

Stop: if this needs live GitHub credentials, changes the hosted API public contract without fixtures, or claims Elixir can be retired.

Delete-First

No daemon, queue scheduler, new store engine, or provider integration. The existing JSON store and Rust review command are enough to prove the lifecycle boundary.

Interface

cerberus-cli hosted-api-worker-fixture --store-state ... --review-id ... --pr-context ... --diff-file ... --out ...

Boundary

The adapter owns state transition rules. The CLI reads inputs, runs review, writes artifacts, and persists the updated store.

Acceptance Oracle

  • A queued review becomes completed with aggregated_verdict.verdict matching the generated artifact.
  • The completed status includes a schema-valid review_run_artifact bound to the hosted request head SHA.
  • Wrong head SHA, missing review id, non-queued status, and stale worker outputs fail closed without mutating the store.
  • Loopback GET /api/reviews/:id reads the completed status after the worker writes state.

Residual

Live GitHub acquisition, production deployment smoke, provider-backed reviewer quality, and Elixir deletion remain separate backlog gates.
