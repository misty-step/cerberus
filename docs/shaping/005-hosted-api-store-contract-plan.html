Backlog 005 / Hosted API Store Lifecycle

Make the Rust hosted API state file a validated contract.

The hosted API worker and local HTTP fixture already persist queued and completed reviews through a JSON state file. This slice names that file format, validates it before use, and keeps the boundary narrow: no SQLite port, no production server, no provider run.

Quality System

Outcome: a local/Sprites/GitHub runner can treat --store-state as a versioned Rust queue/store contract instead of an arbitrary fixture blob.

Proof: failing-first store validation tests, CLI startup rejection of invalid state, worker/server persistence checks, token-leak grep, full gates, and fresh critic.

Stop rule: do not claim production hosted API parity, live GitHub acquisition, provider execution, deployment, or legacy SQLite import.

Current State

HostedApiServiceStoreFixture is structurally deserialized and then trusted. It persists the right local behavior, but the file has no named schema version and no fail-closed check for token-shaped fields.

Chosen Design

Introduce HostedApiReviewStore with schema_version = hosted-api-review-store.v1, keep the old fixture type name as a compatibility alias, and validate loaded/written state in the adapter.

Acceptance

Unsupported versions, invalid review ids, invalid statuses, and raw token fields fail before ready files or worker outputs are written. Existing legacy fixtures without a version continue to parse as v1.

Verification

Claim: the local hosted API queue/store evidence now has a stable, reviewable Rust contract.

Falsifier: arbitrary malformed store JSON starts the server, worker persistence writes invalid state, checked fixtures drift, or secrets serialize into state/report output.

Delete-First

Requirement questioned: adding a database abstraction, storage trait, or queue runtime for one local proof path.

Deleted or simplified: keep one concrete store struct and one state file, with validation as the deep interface.

Only then automated: repeated hosted lifecycle smoke now needs a named format and hard validation.

Review Focus

Attack backwards compatibility, token leakage, overclaiming production parity, and whether validation rejects the wrong things or misses the fields that matter.
