CERBERUS / HARNESS ADAPTER

Let external reviewer commands plug into `ReviewHarness` without entering core semantics.

Build a command adapter in `cerberus-adapter`: write an input envelope, launch a configured command with input/output paths, parse a reviewer artifact, and let core validate or degrade the result.

CHOSEN DESIGN

Subprocess execution lives in `cerberus-adapter`; `cerberus-core` keeps only the trait, artifact validation, aggregation, and degradation semantics.

QUALITY SYSTEM

Standards: no paid calls in tests, timeout is structured, non-zero exits carry stderr, and malformed artifacts still flow through core validation.

STOP IF

The adapter needs provider credentials, harness-specific prompt logic, hosted API changes, or shell execution inside `cerberus-core`.

INPUT

`ReviewerConfig` plus `ReviewRequest.v1` serialized to a temporary command input envelope.

OUTPUT

A `ReviewerArtifact.v1` written by the command to an explicit output path and returned through `ReviewHarness`.

PROOF

Fixture command success, failure, and timeout all exercise the real subprocess path and feed `review_with_harness`.

IMPLEMENTATION FLOW

01

Adapter module

Add `CommandHarness` in `cerberus-adapter` with timeout, stderr capture, and temporary input/output files.

No command execution inside core.

02

Fixture command

Add a deterministic fixture script that emits a reviewer artifact, exits non-zero, or sleeps for timeout tests.

No paid provider calls.

03

Integration tests

Run the command adapter through `cerberus_core::review_with_harness` for success and degraded reviewer paths.

Core remains the artifact acceptance gate.
