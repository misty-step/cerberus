Backlog 015 · Peer Harness Boundary

Make live peer execution inspectable before making it executable.

The runner already renders prompts and parses transcript fixtures. The next deep-module step is a schema-valid execution plan that resolves the peer command, prompt mode, environment requirements, markers, timeout, and unsupported boundaries without invoking Pi, Goose, OpenCode, OMP, or OpenRouter.

Target Outcome

`cerberus-peer-harness --execution-plan-output ...` writes `PeerHarnessExecutionPlan.v1` for any validated profile and input. Operators can inspect the live command contract before enabling paid execution.

Chosen Design

Add one schema artifact and one runner flag. Do not add a new executor, provider client, background job, budget system, or raw harness wrapper.

Verification System

CLI driver writes an execution plan, schema validation checks it, unit tests cover environment and prompt placeholder behavior, and a fresh critic checks the public workflow.

Stop Rule

Stop before live provider invocation. A plan artifact plus fail-closed live mode is enough for this slice.

Quality System

Part Decision
Claim Peer harness live execution can be reviewed as data before any paid call runs.
Falsifier The plan omits required env, leaves `{model}` unresolved, leaks the full prompt into argv unintentionally, or lets live mode call a peer CLI.
Driver `cerberus-peer-harness --execution-plan-output tmp/peer-runner-execution-plan.json`.
Grader `cerberus-cli validate` plus tests for env availability, unresolved model placeholders, prompt placeholder count, and live-mode fail-closed behavior.
Evidence Packet `tmp/peer-runner-execution-plan.json` and command transcript in the backlog receipt.
Critic Fresh-context review gets only the diff and oracle, with the works-critique card.
