BACKLOG 005 / ELIXIR VERDICT STORE RETIREMENT DECISION

Do not port the legacy SQLite verdict store into Rust.

The Elixir store is a runtime-internal queue, verdict, event, and cost database. The Rust contract is the source-agnostic ReviewRunArtifact.v1 receipt plus explicit hosted API fixture state. SQLite compatibility is not required for Rust cutover unless a real consumer depends on reading historical Elixir databases.

Chosen Design

Keep Rust persistence as immutable validated JSON receipts through FileReviewRunArtifactStore. Keep hosted API fixture state as JSON. Do not add a Rust SQLite schema, migration layer, or compatibility importer in this backlog slice.

Quality System

Risk tier: retirement boundary decision. Proof: source audit, focused Elixir store tests, Rust artifact-store tests, machine inventory validation, full gates, and fresh critic on whether the decision overclaims parity.

Elixir Surface

Cerberus.Store owns review_runs, verdicts, events, and review_costs tables plus model-performance queries.

Rust Surface

ReviewRunArtifact.v1 contains verdicts, findings, reviewer artifacts, stats, coverage, reserves, and cost summaries, then validates replay consistency.

Acceptance

The retirement inventory records that SQLite compatibility is intentionally not a cutover requirement, while production hosted queue/store work remains pending.

Stop If

A current caller reads Elixir SQLite directly, historical database import is required for production cutover, or Rust artifact receipts cannot answer review evidence needs.

Verification System

Claim: Rust cutover does not need SQLite compatibility before the Elixir verdict store can be retired through the remaining production queue/store gates.

Falsifier: active docs, code, tests, or callers treat the SQLite tables as a public artifact or require Rust to replay historical *.sqlite3 databases.

Driver: search for SQLite/store usage, compare Elixir store tests with Rust artifact-store/schema tests, and validate the retirement inventory.

Grader: the inventory states the rejected compatibility surface without marking full parity complete, and gates keep both old and new surfaces healthy.

Tradeoff

Deleted requirement: a second Rust persistence schema that mirrors Elixir's normalized tables.

Retained requirement: production Rust hosted queue/store lifecycle still needs a real persistence choice before API cutover.

Deferred only on evidence: historical SQLite import can become a future backlog item if an operator names a concrete database and consumer query.

Execution

1 audit

Read cerberus-elixir/lib/cerberus/store.ex, store tests, Rust artifact-store code, schema validation, API docs, and retirement rows.

Do not infer from old walkthroughs without live code evidence.

2 decide

Classify legacy SQLite as internal compatibility state, not the cross-harness evidence contract.

Keep the Elixir store pending until production Rust hosted queue/store lifecycle is proved.

3 codify

Update the inventory, readable table, docs index, and backlog receipt with the no-SQLite-bridge decision.

Avoid claiming production API parity or deleting legacy files.

4 verify

Run focused Rust and Elixir store gates, retirement validation, full workspace gates, and a fresh critic.

A blocker is any stale next action or parity claim that suggests SQLite was fully replaced.
