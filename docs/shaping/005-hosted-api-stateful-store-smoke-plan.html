Backlog 005 / Hosted API Store Lifecycle

Prove POST-created reviews can be replayed by Rust over HTTP.

The hosted API fixture server can serve static status fixtures. This slice adds a small mutable local store file so a POST-created review ID survives into a later GET request without Elixir, SQLite, queue workers, or provider execution.

Quality System

Outcome: a local/Sprites/GitHub runner can smoke the hosted API queue/store lifecycle with one bounded Rust process.

Proof: real TCP POST then GET against hosted-api-serve-fixture --store-state, persisted JSON inspection, token-leak grep, focused tests, full gates, and fresh critic.

Stop rule: do not claim production hosted API parity, live GitHub acquisition, reviewer execution, deployment, or SQLite compatibility.

Current State

elixir-http-api remains pending because Rust has fixture responses and a bounded listener, but no mutable queue/store lifecycle.

Chosen Design

Add --store-state <path>. The existing --store fixture remains read-only. Stateful mode writes a safe HostedApiServiceStoreFixture JSON document after successful POST creation.

Acceptance

POST /api/reviews returns a review id, GET /api/reviews/<id> returns the queued legacy-shaped record, and the persisted state contains no API key or request token material.

Verification

Claim: Rust can persist and replay a local hosted API review status over HTTP.

Falsifier: the review id is missing on GET, the state file does not persist, static fixture behavior regresses, or secrets serialize.

Delete-First

Requirement questioned: full database or queue parity for this slice.

Deleted or simplified: no SQLite clone, no worker, no provider call, no async runtime.

Only then automated: repeated HTTP smoke needs one durable local state file.

Review Focus

Attack compatibility shape drift, accidental mutation of checked fixtures, token leakage, and any overclaim that this replaces production Elixir.
