Backlog 017 | Peer Harness Boundary

Move provider prompts out of argv.

Add a private prompt-file handoff so Cerberus can run provider-backed peer profiles later without exposing rendered diffs in process arguments. Prove it with a local fixture first; do not spend provider budget in this slice.

Chosen Design

Add one prompt-file mode to the existing profile schema. The runner writes the prompt to a private temp file, substitutes only the path, and removes the file after execution.

Delete-First

No provider client, scheduler, retry layer, upload abstraction, or eval engine. Existing peer profile templates are enough.

Stop Rule

Fixture prompt-file live run passes, provider profiles no longer use {prompt}, full gates pass, and a fresh critic finds no blocking leak or lifecycle issue.

Verification System

  • Claim: prompt text travels through a private file path, not argv.
  • Falsifier: rendered diff appears in args, temp file persists, permissions are broad, or provider profile still trips the argv gate.
  • Driver: CERBERUS_PEER_HARNESS_LIVE=1 cerberus-peer-harness --harness fixture-live-file ...
  • Grader: validated artifact, captured transcript, execution-plan inspection, schema tests, temp-file cleanup assertion.

Critic Focus

Attack prompt leakage, file permissions, cleanup on failure, provider-template truthfulness, and schema compatibility.

Current Evidence

Local CLI help shows: Goose supports --instructions FILE, OpenCode supports --file, and Pi/OMP accept @files in message arguments.

In Scope

Schema mode, runner temp-file lifecycle, fixture profile/script, provider profile templates, docs, tests, QA.

Out Of Scope

Provider spend, model scoring, hosted API parity, retries, and generalized attachment systems.
