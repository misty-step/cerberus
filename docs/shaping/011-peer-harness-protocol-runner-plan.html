Backlog 011

Peer Harness Protocol Runner

Implement the Rust `cerberus-peer-harness` command that the 010 profiles point at, proving the file protocol without spending model budget.

Date: 2026-06-18

Branch: shape/rust-review-engine-backlog

Stop: if the runner needs live peer output parsing to produce a truthful artifact.

Target Outcome

`CommandHarness` can launch `cerberus-peer-harness --harness <id> --input <path> --output <path>` and receive a schema-valid `ReviewerArtifact.v1` for Pi, Goose, OpenCode, or OMP.

Design

Add a second Rust binary in `cerberus-cli`. It validates the profile fixture, selects the requested harness, reads `CommandHarnessInput`, and writes an offline degraded artifact with exact request coverage.

Acceptance Oracle

  • `cargo test --workspace peer_harness_runner`
  • `cargo test -p cerberus-cli --test peer_harness_command`
  • `cargo run --locked -p cerberus-cli --bin cerberus-peer-harness -- --harness pi --input ... --output ...` through a focused fixture or test.
  • `cargo run --locked -p cerberus-cli -- validate <output artifact>` validates the emitted JSON.
  • Full Rust, shell, Node, and legacy Elixir gates remain green.

Protocol

cerberus-peer-harness \
  --harness pi \
  --input /private/tmp/cerberus-command/input.json \
  --output /private/tmp/cerberus-command/output.json

Input:  CommandHarnessInput { reviewer, request }
Output: ReviewerArtifact.v1 {
  status: "degraded",
  verdict: "SKIP",
  coverage.files_reviewed: request.change.files[*].path,
  degraded_reason: "offline protocol runner; live peer execution disabled"
}

Live Call Guard

The runner must not invoke Pi, Goose, OpenCode, OMP, OpenRouter, or paid providers in this slice. A live-mode env flag should fail closed with a clear error until transcript capture and parser tests exist.

Files

  • `crates/cerberus-cli/Cargo.toml` for the new binary target.
  • `crates/cerberus-cli/src/peer_harness.rs` for protocol logic and tests.
  • `docs/shaping/peer-harness-protocol-runner.md` for the design receipt.
  • `backlog.d/011-peer-harness-protocol-runner.md` for sequencing.

Critic Focus

  • Does the output artifact pass the same schema and core identity/coverage checks as a real harness?
  • Can profile paths, harness ids, or env vars accidentally trigger live provider work?
  • Does adding `cerberus-adapter` to the CLI crate create an ownership cycle or push shell semantics into core?
