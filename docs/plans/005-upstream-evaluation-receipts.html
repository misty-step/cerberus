Backlog 005 / Upstream Evaluation Receipts

Emit scoring-ready review receipts without owning evals.

Cerberus should let Daedalus and other labs compare runs by request digest, artifact digest, context tier, harness, model, usage, latency, and validation outcome. The runtime behavior stays the same; the new surface is a redacted receipt bundle emitted beside review outputs.

Chosen Design

Add: `ReviewReceiptBundle.v1` as a separate Rust schema and builder.

Use: existing `ReviewRun`, execution plan, transcript path, and validated artifact.

Avoid: benchmark runners, model promotion logic, transcript excerpts, or provider-specific eval scoring.

Standards

Deterministic digests, redacted paths, no secret values, schema-tested JSON, CLI-compatible defaults.

Acceptance

`review` and `review-pr` can write `ReviewReceiptBundle.v1`; fake OpenCode exposes model/usage metadata; docs define Daedalus handoff.

Verify

`cargo test --locked`, receipt redaction tests, fake fixture/OpenCode/OMP runs, `./scripts/verify.sh`, reviewer re-check.

Stop If

Receipt generation changes review behavior, leaks temp prompt/request paths, or starts implementing evaluation policy.

Change Shape

  • Add `src/receipt.rs` with schema, capability tier derivation, validation outcome, and builder tests.
  • Add telemetry to `ReviewRun` so OpenCode metadata can flow to receipt bundles.
  • Write receipt bundles from `review` and `review-pr`, with optional CLI path override.
  • Update `scripts/verify.sh` to assert `target/cerberus/receipts/*.json` evidence and redaction.

Review Focus

  • Does the bundle contain enough to match two runs by request digest and compare model/harness cost/latency?
  • Can a temp prompt path, request file path, env name, or transcript excerpt leak?
  • Is this still a receipt writer, not an evaluation framework?
