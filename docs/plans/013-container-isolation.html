Backlog 013 / Container Isolation · the security boundary for untrusted-PR review · rewritten after adversarial critique

For untrusted code, the key must never enter the container.

Reviewing a consumer's untrusted third-party PR means the diff is attacker-controlled and can prompt-inject the agent's shell. The hard, counter-intuitive result a security critic forced: an egress allowlist does not protect the credential while the key is in the container — DNS bypasses a CONNECT proxy, and the one host you must allow (the model API) is itself a read/write exfil sink when the agent holds the key. So the boundary is credential-out-of-container, with network and output controls as defense-in-depth. Cerberus ships this as a reference profile behind the substrate seam; trusted self-review stays on the local harness.

Chosen Design

Floor (untrusted-capable): a key-injecting reverse proxy outside the container — OpenCode reaches the model through it; the key never enters the container. Plus: no in-container resolver, internal Docker network (proxy is the only route), and a .git-less throwaway tree (not a linked worktree).

Defense-in-depth: domain egress-allowlist at the proxy + structural validation of emitted artifact fields. Not the boundary — layers on it.

Unchanged: public schema; trusted self-review keeps the local/worktree harness (key-in-env is fine for our own code).

Why "egress allowlist + key in container" is false security (the critic's core finding).
  • DNS tunnel: a CONNECT proxy governs only TCP the agent routes through it. The agent encodes the key into DNS QNAMEs (<hexkey>.evil.com) and the resolver leaks it — the proxy never sees it.
  • The allowed endpoint is the sink: with the key in hand, the agent makes a legitimate HTTPS call to the model API carrying the key in the prompt, then reads it back from the attacker's own account using the same stolen key. The one host the allowlist must permit is itself an exfil channel.
  • Output scrub is unwinnable as a denylist: rev, gzip|base64, char-split across two findings, or just echo $KEY > fix.patch all defeat a transform-matcher. (Verified: no value-scrub exists today, only redact_prompt_path.)

Conclusion: only removing the key from the container closes "use the key to attack the key's own API." Everything else is depth.

Premise resolved

VISION: consumers own the substrate → Cerberus ships a reference container profile behind the seam (swappable) and owns the credential primitive. Trusted vs untrusted is a real split: only the key-out container is "untrusted-capable".

Acceptance

A red-team fixture family (DNS exfil, endpoint-as-sink, output encodings, file-write, worktree escape) all fail to surface the key; a normal container review still yields a valid artifact.

Verify

Assert: the key is absent from the container; zero UDP/53 and zero non-proxy egress; mounted tree has no .git; host checkout digest unchanged; artifact valid.

Stop / waiver

Nothing markets the local or key-in-container path as "safe for untrusted." The "untrusted-safe" label attaches only to key-out + no-resolver + clone-not-worktree, proven by the fuzzed fixture family.

Change Shape (phased)

  • M1 — harness + red-team fixtures + isolated tree. Add a container-opencode substrate (new CommandSubstrateConfig variant) wrapping opencode run in docker run on an internal network. Mount a .git-less throwaway tree (git archive | tar -x of the head SHA), never a linked git worktree; host checkout never mounted. Pinned Dockerfile (opencode + git + ripgrep + ast-grep). Build the red-team fixture family — the proof loop — first, with the substrate failing closed (no model access yet).
  • M2 — key-out reverse proxy (the floor). A sidecar holds the key; OpenCode points at it (provider base-URL + a trusted CA, or a container-scoped dummy token) and the sidecar injects real auth and forwards to the model. No OPENROUTER_API_KEY in the container env. No in-container resolver (static /etc/hosts → sidecar; --dns none). This is what makes the substrate untrusted-capable.
  • M3 — defense-in-depth. Sidecar egress-allowlists the model domain only (belt over the key-out boundary); structural validation of emitted artifact fields (length/grammar/entropy reject, not a secret-denylist); value-scrub of any other allowed_env secrets as backstop.

Trusted self-review is untouched — it stays on the local/worktree harness where key-in-env is acceptable.

Verification System

Claim: a prompt-injected agent reviewing an untrusted PR in the container cannot obtain or exfiltrate the model credential by any vector, because the credential is never in the container; a normal review still produces a valid artifact.

Falsifier: any red-team fixture surfaces the key (network, DNS, endpoint-sink, output, written file); OR the key is found in the container env/fs; OR the mounted tree carries .git; OR the host checkout mutates; OR a clean review can't produce a valid artifact.

Driver: the red-team fixture family through --harness container-opencode, with network capture (no UDP/53, only model egress) + an in-container key-absence probe + an output key-scan.

Grader: network log + key-absence probe + key-scan (literal & transforms) + host-tree digest + artifact validation. A single canned fixture is insufficient — the family must include encoding/DNS/endpoint variants and grow on each near-miss.

Evidence packet: network capture, the fixture family, the key-absence probe output, and the container execution plan, under target/cerberus/container-*.

Cadence: gated in verify.sh when Docker is present; on every change to the container/proxy/mount path.

Gaps / waiver: the "untrusted-safe" claim attaches ONLY to M2+M1 (key-out + no-resolver + clone-not-worktree). M3 is depth. The model-domain pin must track the provider's real hosts.

Milestones

M1 · Harness
Red-team fixture family + container-opencode plumbing (internal network, .git-less tree mount, host not mounted), failing closed. The proof loop, first. Critic on fixture realism + escape vectors.
M2 · Key-out
Key-injecting sidecar (key never in container) + no in-container resolver. The floor that earns the "untrusted-capable" label. Gate: key-absence probe + zero-UDP/53.
M3 · Depth
Domain egress-allowlist at the sidecar + structural artifact-field validation + backstop scrub. Layers on the boundary; never sold as the boundary.

Adversarial Review (run during shaping; folded in)

A fresh-context security critic dismantled the original "egress-allowlist + key-in-container" design: DNS bypasses the proxy; the allowlisted model endpoint is itself an exfil sink while the agent holds the key; a transform-denylist scrub is unwinnable; the mounted git worktree is a live handle to the host .git (verified). The plan was rewritten so key-out is the floor (not optional), the mount is a .git-less clone, the resolver is removed, and acceptance is a red-team family. Residual focus for /deliver: the sidecar TLS/CA story (base-URL override vs trusted CA injection) and confirming OpenCode honors a custom provider base-URL; verify Docker "internal network" truly has no egress but the sidecar.
