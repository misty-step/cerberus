Backlog 016 / Review Emission · supersedes the "structured output" and "custom MCP" framings

Stop fighting the model into a format. Emit, validate, re-ask.

Today the agent must type the whole review as one JSON blob in its final message, and a defensive marker/XML/raw parser tries to recover it (it has failed twice live). Three drafts converged — structured output (OpenCode can't), a custom MCP tool surface (overkill for one agent we control) — onto the simplest thing that works: the agent writes the artifact to a file with its existing write tool, Cerberus validates it, and on failure continues the OpenCode session with the exact error and asks it to fix. The caller-neutral ReviewArtifact.v1 stays; post.rs keeps posting. No MCP, no new dependency. This change deletes more than it adds.

Chosen Design

Emit: agent writes ReviewArtifact.v1 JSON to <workspace>/review-artifact.json (its existing write tool); harness reads + serde-parses it.

Re-ask: validate against the request; on miss/invalid, opencode run --session <id> "<exact error>, fix and rewrite", ≤2 retries.

Delete: the marker/XML/raw extraction (extract_marked_artifact et al.) + the ARTIFACT_BEGIN/END markers + the ~1800-char prompt lecture.

Unchanged: the artifact contract, validation.rs, and post.rs (Cerberus still posts; agent never touches GitHub).

Why keep the neutral artifact (it is not premature). A thin typed data object is Fowler's explicit YAGNI exemption — cheap even at one consumer, testable by value, no GitHub mock. It is exactly the layer that gives idempotency + validation for free — the very things direct-posters had to rebuild (reviewdog/Danger built dedup because naive posting stacks duplicates; GitHub's own review MCP silently drops comments on bad line numbers). Every mature reviewer (PR-Agent — 9 providers behind one interface; CodeRabbit; Greptile) produces-then-posts and treats the host call as a dumb sink. We are already in good company; we keep `post.rs`.
What we deliberately defer / reject. The multi-destination adapter framework waits for a real second consumer (rule of three — we have one GitHub adapter, that is plenty; building the abstraction now is the textbook wrong-abstraction). We do not hand the agent a GitHub-writing tool (loses idempotency/validation and makes Cerberus GitHub-only). And the custom MCP is dropped: for a single agent we control, MCP and write-a-file terminate at the same validator — MCP's only edge is cross-process reuse we don't need yet.

Standards

Per ADR 0003: the artifact contract + validation stay deterministic (oracle-checkable); reliable emission is harness engineering (re-ask), not a quality gate. Agnostic: the agent never knows the destination.

Acceptance

A malformed-then-corrected emission fixture proves the re-ask loop recovers; the live dogfood self-review produces a valid, posted artifact with the old parser deleted.

Verify

./scripts/verify.sh green (fixture + OMP paths moved to file emission); a unit test drives the re-ask loop; one real review-pr dogfood.

Stop / waiver

If OpenCode session continuation (--session) proves unreliable for the re-ask turn, fall back to a fresh run that re-states prior findings, or cap at one retry — don't silently accept an invalid artifact.

Change Shape (ordered)

  • M1 — file emission + delete the parser. Slim build_opencode_message (prompt.rs): drop the field-by-field JSON lecture + marker rules; instruct "write your final ReviewArtifact.v1 JSON to <out-path>." Harness reads the file and serde-parses it. Delete extract_marked_artifact / extract_opencode_text_events / extract_json_between_markers / extract_unmarked_artifact_json (harness.rs:969–1090) + the ARTIFACT_BEGIN/END constants. Move the fixture + OMP emission to write the artifact file too (their tests follow). verify.sh green.
  • M2 — validate-and-re-ask loop. Move validate_artifact_for_request into the run loop (today post-hoc at main.rs:365/447). Capture the OpenCode sessionID from the JSON events; on missing/invalid artifact, opencode run --session <id> "<validation error>: fix and rewrite <out-path>"; re-read + re-validate; ≤2 retries, then fail. The loop belongs where request + harness + validator meet (kernel or a thin orchestration layer); the harness gains "continue this session with a message."

Oracle — executable definition of done

## 1. Re-ask recovers (THE test, deterministic)
# a fixture "agent" that writes an INVALID artifact on attempt 1 and a VALID one on attempt 2
# -> the review succeeds; the transcript shows the re-ask carried the exact validation error.

## 2. Parser is gone
rg -n 'extract_marked_artifact|ARTIFACT_BEGIN|extract_unmarked_artifact_json' src/   # -> no production hits

## 3. No regression + no new deps
./scripts/verify.sh            # green (fixture + OMP paths on file emission)
git diff Cargo.toml            # no new dependency

## 4. Live dogfood (operator-gated)
cargo run -- review-pr --number  --repo misty-step/cerberus --model openrouter/z-ai/glm-5.2 \
  --allow-env OPENROUTER_API_KEY --timeout-seconds 1800 --post
# valid artifact posted; if the first emission was invalid, the receipt/transcript shows a recovered retry.

Verification System

Claim: the agent reliably yields a valid ReviewArtifact.v1 via file emission + a bounded validate-and-re-ask loop, with the defensive parser deleted and no new dependency.

Falsifier: an invalid first emission is silently accepted; the re-ask doesn't carry the real validation error; verify.sh regresses; a new dep appears; the agent can reach GitHub directly.

Driver: the malformed-then-corrected fixture through the review path (deterministic) + one real review-pr dogfood.

Grader: the fixture review succeeds only via the retry; rg shows the parser gone; verify.sh exit 0; the dogfood posts a valid artifact.

Evidence packet: the fixture + its transcript showing the carried error; the dogfood artifact/receipt under target/cerberus/.

Cadence: the fixture in verify.sh always; the dogfood on prompt/emission change.

Gaps/waiver: faithfulness/quality is out of scope (that's 015); this is about *valid structure*, reliably.

Milestones

M1 · Emit
File emission + delete the parser/markers + slim the prompt + move fixture/OMP to file emission. Net-neutral on reliability, big on simplicity. Gate: verify.sh green, parser gone.
M2 · Re-ask
Validate-in-loop + OpenCode session-continue retry (≤2) carrying the exact error. The reliability win. Gate: malformed-then-corrected fixture recovers.

Adversarial Review Focus

For /deliver's diverse-provider review (artifact + oracle only): (1) Does deleting the marker/XML/raw parser lose a real recovery path the file approach doesn't cover (e.g., the agent writes the file and chatters JSON)? (2) Is the re-ask loop bounded and fail-closed — can it ever silently accept invalid, or loop unbounded on a stubborn model? (3) Does moving validation into the loop change any existing behavior at main.rs:365/447 (degraded/skip lifecycle, posting decisions)? (4) Did the fixture/OMP move to file emission actually keep verify.sh's adversarial cases (overstated tier, duplicate artifact) meaningful?
