Backlog 018 / Agent-native handoff · the primary form factor, made first-class

Spawn Cerberus. Read the review. Branch on the exit code.

The way Cerberus is actually used on our own repos: an agent working in the repo dispatches it as a subprocess, gets a review of the current changes, and decides what's blocking. The engine already produces the review — cerberus review → ReviewArtifact.v1 + render_markdown. Two gaps make it clumsy: (1) a FAIL verdict still exits 0, so an agent can't gate without parsing; (2) reviewing a local range is a two-command dance through a temp request file. Close both: a verdict-gated exit code, and one no-GitHub command that hands the review back on stdout. No token, no posting — that's the consumer's layer (014).

Chosen Design

Gate: --fail-on <none|warn|fail> on review + review-diff. Exit 0 proceed · 1 blocking verdict · 2 Cerberus error.

Entry: cerberus review-diff --base <ref> [--head HEAD] fuses build_git_range_request + kernel + render; review markdown to stdout by default.

Reuse: no new review logic — request-builder, ReviewKernel, render_markdown unchanged.

Out of scope: GitHub/posting (014); MCP wrapper (later); uncommitted working-tree diff (follow-on).

Why exit codes, not just the artifact. Prior art is unanimous — reviewdog -fail-level, linters, Claude Code hooks: an orchestrating agent needs two channels, the full machine-readable review and a one-bit block/proceed signal. The load-bearing subtlety: a blocking verdict (a successful review that found issues) must be distinguishable from a tool error (no review produced). Today both would collide on exit 1, so this moves tool errors to exit 2 and reserves exit 1 for "blocked — read the review."

Standards

Caller-neutral artifact + a host-neutral handoff. No GitHub, no token in this path. ReviewRequest.v1 → ReviewArtifact.v1 and validation.rs untouched; existing review stays back-compatible without --fail-on.

Acceptance

A fixture exit-code matrix in verify.sh (PASS/WARN/FAIL × --fail-on × error path) plus one live review-diff on a real branch returning review-on-stdout + a branchable code.

Verify

./scripts/verify.sh green; each fixture's $? equals the matrix; stdout carries only the review; review-diff --help + a README handoff section exist.

Stop / waiver

If "diff" misleads toward uncommitted changes, rename review-diff → review-range. Keep this a dumb handoff — measuring review quality is Daedalus, not here.

Exit-code contract (load-bearing)

ExitMeansWhen
0clean / proceedvalid artifact; verdict below --fail-on (or none, default)
1blocking — read itvalid artifact; verdict ≥ threshold (fail⇒FAIL; warn⇒WARN/FAIL)
2Cerberus errorharness/validation/IO failure; untrusted lifecycle; no valid artifact

SKIP ("couldn't review") is not blocking ⇒ exit 0 unless explicitly gated; the agent reads the verdict field for nuance. Today main() returns Result (Err⇒1); introduce explicit process::exit codes.

Adversarial Review Focus

For diverse-provider review (artifact + oracle only): (1) Can a blocking verdict and a tool error ever collide on the same exit code? (2) Does --fail-on none truly preserve existing review exit behavior (no silent break of current callers / verify.sh)? (3) Is review-diff stdout genuinely review-only under a real OpenCode run (no log/telemetry bleed)? (4) Does review-diff reuse the request-builder + kernel verbatim, or did it fork review logic?
