# Isolate untrusted-PR review: scoped ephemeral credential + light container

Priority: P1 · Status: in-progress (M1 done, live-verified 2026-07-01; M2 done, live-verified 2026-07-01; M3 not started) · Estimate: L · Factory epic: 4 · Shape: docs/plans/013-container-isolation.html

## Goal
Let a fully-capable Cerberus agent review UNTRUSTED third-party PRs safely by breaking the lethal trifecta at the cheapest leg — make the model credential *worthless* (per-review, capped, revoked) rather than hiding a powerful one — with a lightweight container for non-credential damage.

## Non-Goals
- Not a hosted multi-tenant service (local/CI spawns a container behind the same kernel).
- Not for trusted self-review (that stays on the local/worktree harness).
- Never call the local/un-scoped path "untrusted-safe."

## Oracle
- [x] Each untrusted review mints a per-review OpenRouter key with a ~$5 USD `limit` (wiggle-room starting value, right-sized later from cost telemetry) via the provisioning API, injects it, and `DELETE`s it on teardown. **Verified live 2026-07-01** — see M1 live-verification below.
- [x] **Steal-and-replay:** a key captured mid-run FAILS after teardown (401 revoked / 402 over-cap); the minted key GET returns 404/disabled. **Verified live 2026-07-01.**
- [x] **Crash safety:** killing the run mid-review leaves no usable key — an orphan-key sweeper revokes review-tagged keys on the next run. **Verified live 2026-07-01.**
- [x] The container blocks non-model egress (`curl evil.com` fails) while allowing exactly the model API, mounts a `.git`-less `git archive` tree (not a worktree), never mounts the host checkout; host digest unchanged. **Verified live 2026-07-01 — squid CONNECT-proxy allowlist (M2 slice 2); non-model host confirmed blocked (`curl` exit 7 against a real public host), model API confirmed reachable (real authenticated chat completion, `200`).**
- [x] A normal review still yields a valid `ReviewArtifact.v1`; `verify.sh` green (container path skipped without Docker).

## Verification System
- Claim: a prompt-injected agent on an untrusted PR cannot obtain a credential worth stealing or reach a non-model host; any credential it captures is capped and revoked by run-end.
- Falsifier: the minted key outlives the run; the key has no cap; a stolen key still works post-run; the agent reaches a non-model host; the host checkout mutates; a clean review can't produce a valid artifact.
- Driver: steal-and-replay probe + crash-injection (kill mid-run → sweeper revokes) + red-team container family with egress capture.
- Grader: post-run the minted key is dead (API) and was capped; egress log model-only; host-tree digest unchanged; artifact validates.
- Evidence packet: key create/delete transcript, steal-and-replay result, crash-sweeper log, egress capture under `target/cerberus/container-*`.
- Cadence: key-lifecycle tests always; container red-team gated on Docker in `verify.sh`; on every change to the credential/sandbox path.
- Gaps/waiver: residual = a stolen key usable for seconds up to the cap until DELETE; accepted. Provisioning key is secret-zero (host-side). Provider lock-in → option B (key-out proxy) fallback.

### M1 status (2026-07-01) — production-enabled, live-verified
Implemented in `src/openrouter_keys.rs` (+ CLI wiring in `src/main.rs`, `review`/`review-diff`/`review-pr`):
- `ProvisioningClient::{mint_key, revoke_key, list_keys}` against OpenRouter's provisioning API (POST/GET/DELETE `/api/v1/keys`), `ScopedKeyGuard` (RAII, `Drop` revokes on any early return or panic within-process, idempotent on double-revoke), `sweep_orphaned_keys` (name-prefix + age, `cerberus-review-<unix-ts>-<tag>`), `mint_review_key` (sweep-then-mint entry point).
- CLI: `--openrouter-scoped-key` (off by default, trusted self-review unaffected) + `--openrouter-provisioning-key-file`/`-env` (explicit-only, no ambient fallback — house `--gh-token-file`/`-env` pattern) + `--openrouter-key-limit-usd` (default 5.0) + `--openrouter-orphan-sweep-seconds` (default 1800).
- 13 unit tests against a hand-rolled TCP mock server (no new test-infra dependency) cover mint/revoke/list/sweep/guard-drop/double-revoke/sweep-then-mint ordering/sweep-failure-doesn't-block-mint. `verify.sh` covers the CLI-level explicit-source refusal (no source / both sources) end-to-end through the real binary.

**Live-verification evidence (2026-07-01, against the real OpenRouter provisioning API via `OPENROUTER_MANAGEMENT_KEY`, out-of-repo scratch harness, no code changes):**

1. *Steal-and-replay.* Ran the real `cerberus review --openrouter-scoped-key` CLI with `--harness opencode` pointed at a throwaway substrate script that exfiltrates `OPENROUTER_API_KEY` into a file (simulating a prompt-injected agent with env access) and holds an 8s window before failing.
   - Mid-run, while cerberus was still alive and the guard not yet dropped: the exfiltrated key answered a real chat completion — `HTTP 200`, `id: gen-1782952910-9eg8sGNGRTnOGt7j6vjs`.
   - After the run exited (review failed → `?` propagated → `ScopedKeyGuard::drop` fired in-process): the same key replayed against the same endpoint — `HTTP 401 {"error":{"message":"User not found.","code":401}}`.
   - Separately, a direct mint→GET→DELETE→GET round trip against `/api/v1/keys` confirmed the "GET returns 404" half of the oracle line: pre-delete GET → `200, disabled:false`; DELETE → `200 {"deleted":true}`; post-delete GET → `404 {"error":{"message":"API key not found","code":404}}`; a second DELETE on the same hash also returned `404`, confirming the idempotent-404-as-success assumption baked into `revoke_key`/`ScopedKeyGuard`.
2. *Crash safety.* Same substrate script, `LEAK_HOLD_SECONDS=25`. Once the key was captured, sent `kill -9` to the cerberus **parent** process (not the child) — this bypasses Rust unwinding entirely, so `ScopedKeyGuard::drop` cannot run.
   - Immediately after the parent was confirmed dead (`kill -9` + process-exit check), replaying the orphaned key still worked — `HTTP 200, id: gen-1782952978-BIbIqtYrkMXhP5hiZ7Zu` — proving the crash genuinely leaves a live, unrevoked key (the exact residual the sweeper exists for).
   - Ran `cerberus review --openrouter-scoped-key --openrouter-orphan-sweep-seconds 1` again (a fresh, unrelated review). Stderr logged `cerberus: orphan sweep revoked 1 stale scoped OpenRouter key(s) from a prior run`.
   - Replaying the orphaned key a third time now failed — `HTTP 401 {"error":{"message":"User not found.","code":401}}` — confirming the sweeper closed the crash window on the very next run.
3. *Account hygiene.* Listed `/api/v1/keys` after all probes: zero `cerberus-review-*`/probe keys remained (all three minted keys — leaky-agent run, crash-drill run, direct GET-behavior probe — were cleanly revoked); the account's 10 pre-existing keys were untouched. Total live spend across all three real completions: ~$0.000006 (three `gpt-4o-mini` 1-token completions), well inside the tiny test caps used (`$0.01`–`$0.05`).

No repo code changed for this verification pass — evidence was gathered via the CLI's existing public flags plus direct `curl` calls to `/api/v1/keys` for the GET/DELETE round trip; the probe scripts lived outside the repo. **Verdict: M1 is production-enabled** — safe to wire into a real untrusted-PR deployment (BB epic 5) with `--openrouter-scoped-key`.

### M2 status (2026-07-01) — DONE, live-verified (both slices)
New `src/container.rs`: a `container-opencode` `HarnessKind`/`ReviewSubstrate` variant, wired the same way as `opencode`/`omp`/`fixture` (`--harness container-opencode` in `review`/`review-diff`/`review-pr`).

**Slice 1 (containment floor):**
- **Never mounts the host checkout.** `prepare_archive_workspace` extracts the review's `sha` via `git archive --format=tar <sha> | tar -x` (a host-side subprocess pair, piped) into a disposable per-run temp root; only that extracted copy — never `request.context.workspaces.head.path` itself — is ever named in a `docker run -v` argument. `request_with_container_paths` rewrites the request JSON the agent reads so its `workspaces.head.path`/`workspaces.base.path` point at the container-internal mount (`/cerberus/workspace/...`), never the real host path.
- **`.git`-less by construction.** `git archive` extracts file content only, unlike `git worktree add`'s `.git` *file* (`gitdir: <host>/.git/worktrees/...`) — a direct handle back to the host's object store. Confirmed live below, and separately confirmed by hand that this repo's own `git worktree add` (used by the *local*, trusted-review harness) does leave that handle — the contrast this design exists to avoid.
- `--read-only` rootfs, `--cap-drop ALL`, `--security-opt no-new-privileges`, `--pids-limit 256`, `--rm`, `--user <host-uid>:<host-gid>` (not root — `--cap-drop ALL` removes `CAP_DAC_OVERRIDE`, so container-root no longer bypasses normal permission bits; this is also a hardening win on its own).
- **Timeout handling** mirrors the local harness's poll loop, but a container timeout also issues `docker kill <name>` — killing the `docker run` *client* process alone does not stop the container, which dockerd runs server-side independent of that client.
- **Environment quirk found and handled:** this operator's local Docker (`colima`) only bind-mounts `$HOME` into its VM by default; the OS temp dir (`/var/folders/...` on macOS) is invisible to it, so `-v` mounts of a `tempfile`-created dir silently resolve to *empty* directories inside the container instead of failing loudly. Added `--container-host-root <dir>` (defaults to the OS temp dir, correct on native-Docker CI/Linux) so a VM-boundary Docker context can point the per-run root at something the daemon can actually see (`verify.sh` uses `target/cerberus/container-hostroot`, inside the checkout).

**Slice 2 (model-API-only exception + orphan sweeper), landed same day:**
- **Egress topology:** the review container joins a per-run `--internal` Docker network (no WAN route at all) whose only reachable peer is a `squid` (`ubuntu/squid:latest`) forward-proxy container that also joins the default network. squid's config (`http_access allow CONNECT allowed_connect_host allowed_connect_port` / `deny CONNECT` / `deny all`) allows `CONNECT` to exactly `--container-egress-allow-host` (default `openrouter.ai:443`) and nothing else — read straight off the plaintext `CONNECT` request line, so squid never terminates TLS and never sees whatever credential flows through the tunnel it opens. Chose a well-audited stock proxy over hand-rolled Rust specifically because reimplementing CONNECT-tunnel relaying is the kind of security-sensitive protocol surface backlog 013's round-1 critique warned about; the "non-Rust" surface is one templated config file, not an ongoing codebase.
- **Scoped-key passthrough:** `write_allowed_env_file` forwards `request.policy.allowed_env` (exactly what `--openrouter-scoped-key`, M1, populates) into the container via `--env-file` on a private (0600) file under the already-0700 `host_root`, not `-e NAME=VALUE` — `-e` values are visible in `docker inspect`/`ps` on the host, which a scoped OpenRouter key is exactly the kind of value to keep out of.
- **Orphaned-container sweeper**, parallel to M1's orphan-key sweeper: `sweep_orphaned_container_resources` runs at the start of every `run_container_substrate` call, before this run's own resources exist. Container and network names now carry `<unix-ts>-<8-hex-id>` (down from a full UUID — see the DNS-label-length note below), so age is parsed straight out of the name the same way `openrouter_keys::key_age` does, no `docker ps` timestamp-format parsing needed. `--rm` alone doesn't cover a `SIGKILL`ed `cerberus` process: it only removes a container on its own exit, which dockerd manages independent of whether `cerberus` is still alive.
- **Crash safety within-process:** `EgressProxyGuard` (RAII, same shape as `ScopedKeyGuard`) tears down the proxy container + network on `Drop` — covers a panic or an early `?` return; the sweeper is what covers a `SIGKILL`.
- **Bug found and fixed:** initial proxy/review container names embedded a full UUID, pushing `cerberus-review-<ts>-<uuid>-proxy` past 63 characters — the DNS label length limit (RFC 1035). Docker allowed creating the container but the review container's embedded-DNS lookup of the proxy *by name* failed silently (`Name or service not known`) rather than erroring at creation, so the review container had no route to the proxy at all. Shortened the unique suffix to 8 hex characters (still astronomically collision-safe combined with the second-resolution timestamp) and confirmed resolution works.
- **Also found and fixed:** the allowed-host CONNECT probe's response line ends `\r\n`; `read -r` strips the `\n` but keeps the `\r`, which was leaking an unescaped control character into the results JSON and silently breaking `verify.sh`'s grep match. Stripped it in the fixture script.
- 10 new unit tests in `container.rs` (name/age round-trip for both container kinds, `parse_host_port` validation, squid config shape, env-file forwarding) on top of slice 1's 4.
- Fixture updated: `redteam-container-substrate` now also sends a raw `CONNECT` (bash `/dev/tcp`, no curl) for the allowed host through `$HTTPS_PROXY` and records squid's status line, alongside the five existing escape probes. `verify.sh`'s red-team block asserts on it, plus a new fabricated-orphan block (old-timestamped container + network created by hand, then swept by the next container-opencode run).

**Live-verification evidence (2026-07-01):**

*Via `verify.sh`'s own red-team block, real Docker daemon, cost-free (no provisioning key needed):*
1. Both fixtures (diff-only packet + repo_head-via-`git archive`) still show every escape vector failing exactly as in slice 1 (`dns_lookup_exit_code: 2`, `egress_connect_exit_code: 1`, both write escapes `1`/read-only-filesystem, `dot_git_present: false`), **plus** `"allowed_connect_status": "HTTP/1.1 200 Connection established"` — the one narrow exception works.

*Standalone capstone (M1+M2 integration, real spend ~$0.000002, outside the repo like the M1 pass):* ran `cerberus review --harness container-opencode --openrouter-scoped-key --openrouter-provisioning-key-env OPENROUTER_MANAGEMENT_KEY` with a `curlimages/curl`-based substrate (the minimal `debian:bookworm-slim` fixture image has no curl; this is `sh`, not `bash`, since curl's image has no bash either). No repo code changed for the capstone pass; it exercises only existing public CLI flags. **Verdict: M2 is done and live-verified** — `container-opencode` is now a real, usable untrusted-PR review substrate, not just a red-team target.

## Notes
**Why:** shaped over two adversarial rounds (2026-06-22/23). Round 1 critic killed the egress-allowlist + key-in-container design (DNS bypasses a CONNECT proxy; the allowlisted model endpoint is itself an exfil sink while the agent holds a valuable key; transform-denylist scrub is unwinnable; a `git worktree` mount is a host `.git` handle). Round 2 (premise challenge + research) reframed via the lethal trifecta (Willison): break the *value* leg with a scoped ephemeral key — Anthropic's production pattern for Claude Code sandboxing — which dissolves the round-1 attacks (a worthless key isn't worth tunneling, and can't outspend its cap before revocation). Grounded: OpenRouter provisioning API supports per-key USD cap + DELETE; a 2026-07-01 docs pass suggests `expires_at` may now exist too (unverified live) — see M1 status. Key-out proxy demoted to an option; tool-execution split (C) is the north-star architecture. Trusted self-review unaffected. Full design + design-space table in the linked HTML plan.
