# Grooming Plan — 2026-03-14

## Session Theme: Elixir Migration

**Direction locked:** Migrate Cerberus engine from Python/Shell/GHA to Elixir/OTP.
Converge with Bitterblossom architecture for dual-mode operation (standalone + BB module).

## Strategic Rationale

1. **Cost:** GHA minutes are expensive ($0.008/min × N reviewers × waves). Elixir on Fly.io is ~3x cheaper.
2. **Speed:** Wave escalation adds sequential overhead (5-15 min). Single-pass routed execution: 2-5 min.
3. **Simplicity:** 10.8K LOC Python+Shell → ~2K LOC Elixir (based on BB's 4:1 ratio).
4. **Convergence:** BB already proved Elixir/OTP for agent orchestration (6,506→1,649 LOC, first-try success).
5. **Ecosystem:** ReqLLM (OpenRouter native), Instructor (Ecto schemas), sprites-ex, OTel+Langfuse all production-ready.

## Architecture

```
GitHub PR event
    ├─ Standalone: thin GHA action → POST /api/review → Cerberus on Fly.io
    └─ BB mode: Conductor invokes Cerberus.ReviewRunner

Cerberus.Application (Elixir/OTP)
├── Cerberus.Config           (persona registry, model pools, verdict rules)
├── Cerberus.Store             (SQLite: runs, verdicts, costs, events)
├── Cerberus.Router            (GenServer: PR classification → panel selection)
├── Cerberus.ReviewSupervisor  (DynamicSupervisor)
│   └── Cerberus.Reviewer      (GenServer per perspective — ReqLLM + Instructor)
├── Cerberus.Verdict           (aggregation, finding dedup, merge gating)
├── Cerberus.GitHub            (comments, reviews, check runs)
├── Cerberus.Telemetry         (OpenTelemetry → Langfuse)
└── CerberusWeb.Endpoint       (Phoenix LiveView dashboard + API)
```

## Key Design Decisions

- **No Pi CLI:** Direct OpenRouter API calls via ReqLLM. Eliminates subprocess management.
- **No Sprites for reviewers:** Reviewers make API calls, not code execution. Sprites are for BB's builder phase.
- **No waves:** Single-pass routed panel with reserve escalation on disagreement/low confidence.
- **Tool definitions:** Reviewers get GitHub read tools via OpenAI function calling, not filesystem access.
- **Dual-mode:** Standalone Elixir app + Conductor.Worker behaviour (optional BB dependency).
